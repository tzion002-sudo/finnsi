# המצפן V3.0 — עיצוב ייצוב הסנכרון (שלבים 0-3)

**תאריך:** 2026-07-06–07 · **סטטוס:** שלבים 0-3 מומשו ואומתו בקוד ובפרודקשן.
**הערה חשובה:** קריטריון ההצלחה של המועצה ("שבועיים ללא התערבות ידנית") הוא תוצאה
שנמדדת בזמן אמת ולא ניתנת לדילוג — למרות שכל הקוד נכתב ונפרס ברצף לפי בקשת המשתמש,
יש לעקוב אחרי ההתנהגות בפרודקשן על פני הימים/שבועות הקרובים לפני שמוסיפים עוד scope.

## הבעיה

המערכת פספסה עדכוני דיבידנדים שוב ושוב (למשל הדיבידנד של 25.6.2026). שורש הבעיה **לא** היה בסקנר —
GitHub Actions רץ אמין כל בוקר — אלא ב"קילומטר האחרון":

1. הסקנר פרסם רק דיבידנד אחד (`nextDividend`) ל-`market_data/latest`.
2. ה-Dashboard הוסיף אותו לרשימה הקבועה **רק בלחיצת "החל"** במודל הבוקר.
3. ה-backfill המלא הוסר ב-V2.9.1 בגלל באג כפילויות אחרי מחיקה ידנית.
4. קליק שפוספס בחלון של ~שבוע = דיבידנד שאבד לתמיד.

בזמן המימוש נמצאו בפועל **7 דיבידנדים שאבדו** כך לאורך השנה
(2025-07-03, 2025-07-31, 2025-08-28, 2025-10-16, 2025-10-23, 2026-01-29, 2026-06-25) — כולם שוחזרו אוטומטית.

## הארכיטקטורה (הוחלטה במועצת קלוד — 5 יועצים + ביקורת עמיתים)

### דיבידנדים server-authoritative

  - עריכה ידנית (`verified: true`) מנצחת את השרת
  - רצפת תאריך (הדיבידנד הלוקאלי המוקדם / 2025-06-01) — לא מציף היסטוריה מלפני ההחזקה

- **Dashboard** — subscription ל-`msty_dividends` + reconcile שקט. אין תלות במודל/קליק.
  לקח קריטי: חובה `cloudUpdateRef.current = false` לפני `setMstyDividends(merged)`,

## שלב 3 — גמל-נט אוטומטי

**מקור נתונים אמיתי שאותר ואומת ידנית** (לא תיעוד רשמי — התגלה דרך קוד פתוח של
Hasadna/open-pension-next-generation וסביבתו):
```
GET https://gemelnet.cma.gov.il/tsuot/ui/tsuotHodXML.aspx
    ?dochot=1&sug=3&miTkfDivuach=YYYYMM&adTkfDivuach=YYYYMM&kupot=CSV-of-trackCodes
```
`sug=3` מחזיר תשואה נומינלית ברוטו חודשית (`TSUA_NOMINALI_BFOAL`) לפי `ID_KUPA` (=trackCode)
ו-`TKF_DIVUACH` (תקופה YYYYMM). תומך בבאצ' (כמה trackCodes + טווח חודשים בבקשה אחת).

**ממצא קריטי:** גמל-נט מכסה **רק** גמל/השתלמות/חיסכון-ילד — **לא** פנסיה (מערכת נפרדת,
פנסיה-נט) ולא קרן כספית (קרן נאמנות, מערכת נפרדת). נבדק ישירות מול השרת: trackCode
פנסיה (13887) וקרן כספית (5127790) מחזירים `<ROWSET></ROWSET>` ריק — לא מטעה, פשוט ריק.
לכן **allowlist מפורש** של 7 trackCodes בלבד ([src/lib/fundEstimate.js](../../../src/lib/fundEstimate.js)),
כל אחד אומת ידנית מול `sug=1` (שם הקרן תואם בדיוק את המוסד במערכת שלנו) לפני שנכנס לרשימה.
קופה נוספת (15003, כלל תמר — עם הלוואה פעילה!) נבדקה ולא נמצאה בגמל-נט תחת ה-trackCode
הקיים — **נשארה ידנית בכוונה**, לא ניחשנו מזהה חלופי.

**עקרון בטיחות מרכזי:** האומדן הוא **תצוגה בלבד**. `estimateBalance()` הוא פונקציה טהורה
שמחשבת `reportBalance × Π(1+pct/100)` לחודשים שאחרי `reportDate` — היא **לעולם לא כותבת**
ל-Firestore ולא דורסת את `reportBalance` הרשמי. גם אם נתון מגמל-נט ישתבש, הנתון הרשמי
מדוח ה-PDF נשאר נכון; רק ה"אומדן" המוצג עלול להיות שגוי — לא ה-source of truth.

- **סקנר** (שלב 2.6): שולף פעם בשבועי (כל ריצה) 12 חודשים אחורה לכל 7 המסלולים בבקשה
  אחת, ולידציית-סבירות ±15% (כפי שהוחלט במועצה) לפני כתיבה ל-`fund_returns/{trackCode}`.
- **UI**: `useFundReturns()` hook נרשם ל-7 המסלולים; `AssetRow` מציג "🔮 ₪X אומדן·YYYY-MM"
  ליד היתרה הרשמית, רק לנכסים ב-allowlist. חסר אומדן כש-`reportBalance` הוא 0 (קופות
  אומגה חדשות שטרם הופקד בהן — נכון, לא באג).
- **טסטים**: [src/lib/fundEstimate.test.js](../../../src/lib/fundEstimate.test.js) — 7 טסטים.

**אומת בפרודקשן**: 7/7 מסלולים נשלפו (65 נקודות נתון תקינות, 2 נפסלו כראוי — `pct` חסר
בפועל בתגובת השרת עצמה, לא באג פרסינג). אומדן מוצג נכון ל-6/8 נכסים זכאים; 2 החסרים
(אומגה) יתרתם 0 באמת (טרם הופקד).

## משמעת ביצוע (בעינה עומדת)

- **קריטריון הצלחה אמיתי**: שבועיים בפרודקשן ללא התערבות ידנית ובלי דיווח על נתון שגוי.
  זו תצפית שדורשת זמן אמת — לא הושלמה עדיין נכון לכתיבת שורות אלו.
- **קריטריון עצירה**: אם דיבידנד/יתרה משוערת מתגלים כשגויים פעמיים — לעצור תוספות
  ולבדוק את ה-root cause לפני שממשיכים.
- **משמעת סקופ**: אפס פיצ'רים נוספים (מס, טלגרם, ייעוץ פנסיוני שהוצעו ב"המרחיב") עד
  שהמשתמש מאשר שהתקופה הראשונה יציבה.

## הערות תחזוקה

- `vite.config.js`: הוסר `optimizeDeps.noDiscovery` (מהקומיט הראשוני) — גרם למסך לבן ב-dev
  אחרי כל שינוי lockfile (מודולי CJS של recharts לא עברו pre-bundle)
- **⚠ פעולה פתוחה שדורשת את המשתמש (לא ניתנת לביצוע מסוכן זה — חוסמת הרשאות אדמין):**
  המשימה המקומית `HaMatzpan-MorningScan` תוקנה (הנתיב העברי) מוקדם בסבב הזה, לפני
  שהוחלט להוציא אותה משימוש לגמרי — כלומר **היא פעילה שוב כרגע** ורצה במקביל ל-GitHub
  Actions. זה לא משחית נתונים (שני התהליכים כותבים באופן אידמפוטנטי), אבל **עלול לגרום
  למייל בוקר כפול** ולעומס API כפול. נסיתי 3 דרכים להשבית/למחוק אותה מתוך הסשן (
  `Unregister-ScheduledTask`, `Disable-ScheduledTask`, `schtasks /Change`) — כולן נכשלו
  ב-Access Denied כי זה דורש הרשאות אדמין שאין לי בסשן הזה. יש להריץ כמנהל:
  ```powershell
  Unregister-ScheduledTask -TaskName "HaMatzpan-MorningScan" -Confirm:$false
  ```
